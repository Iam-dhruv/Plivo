System Architecture and Design Summary

This document details the architectural choices, system-level optimizations, and iterative design process used to construct a resilient, ultra-low latency UDP pipeline for "The Flaky Network" challenge.

1. Evolution of the Design

The initial architecture relied on a combination of Forward Error Correction (FEC) and an Automatic Repeat reQuest (ARQ) feedback loop. A dedicated playout thread on the receiver attempted to wake up at exact intervals using clock_nanosleep to deliver frames right at their deadline.

This approach was ultimately abandoned due to system-level constraints. Operating system thread schedulers and local network stack traversal introduced micro-stutters, causing packets to arrive microscopically late and resulting in 100% deadline misses. Furthermore, ARQ (NACKs) proved too slow for ultra-low latency targets, as the round-trip time for a retransmission invariably exceeded the tight playout deadlines.

2. Sender Architecture: Hybrid Redundancy

To eliminate the need for time-costly retransmissions, the sender was redesigned to proactively push redundancy using a hybrid FEC and duplication model.

  • FEC Grouping (N=4): The sender groups frames into blocks of four, computing a bitwise XOR across the payloads. A single parity packet is emitted at the end of the block, capable of mathematically reconstructing any single lost frame within that group.
  • Selective Frame Duplication: The first two frames of every block (frames 0 and 1) possess the most immediate relative deadlines. Waiting for the parity packet at the end of the block to recover them would introduce too much delay. The sender intercepts these two frames and transmits an immediate, exact duplicate for each.
  • Bandwidth Optimization: The assignment strictly enforces a ≤ 2.0x bandwidth limit. By sending 4 data frames, 2 duplicate frames, and 1 parity frame, the system transmits 7 packets for every 4 frames generated. This yields an overhead ratio of 1.75x, efficiently utilizing the available bandwidth budget to maximize survivability without triggering a penalty.

3. Receiver Architecture: Zero-Jitter Delivery

The receiver was entirely overhauled to remove self-inflicted latency. The fixed-delay playout loop and all reliance on absolute system clocks were completely removed.

  • Immediate Forwarding: The moment a valid packet arrives from the network, it is tracked and immediately pushed out to the harness player's socket. This bypasses the OS scheduler entirely, trusting the harness to buffer early arrivals until their official deadline.
  • Exact-Sequence FEC Tracker: The receiver maintains a small, lightweight state tracker for incoming FEC groups. If a gap is detected and the corresponding parity packet arrives, the missing frame is reconstructed using XOR operations and instantly delivered.
  • Out-of-Order Processing: By delivering packets as they arrive or are reconstructed, the receiver ensures that slightly delayed or reordered packets count as soon as possible, rather than artificially holding them in a local jitter buffer.

4. System-Level Considerations

In the C implementation, memory ownership and fast execution are critical. The receiver utilizes tightly packed structs (__attribute__((packed))) to ensure the wire format translates directly to memory without padding issues. Network byte order is strictly enforced using htonl and ntohl for multi-byte sequence headers. By shifting the buffering responsibility to the final harness socket and using immediate pass-through, the system avoids complex thread synchronization (mutexes) and memory locks, resulting in a highly performant, lock-free datapath.
